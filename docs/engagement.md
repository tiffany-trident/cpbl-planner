# 使用者黏著度規劃 Roadmap

目標：讓 CPBL Planner 從「路人查賽程」進化到「球迷不可或缺的個人化工具」。

## 實作狀態

  - 實作細節見 [features.md 個人化功能章節](features.md#個人化功能phase-1)
  - UI 實作紀錄與教訓見 [uidesign-changelog.md](uidesign-changelog.md)
- **Phase 2** — 尚未開始（球場巡禮徽章、現場戰績卡、Streak、年度回顧）
- **Phase 3-6** — 尚未開始

## 視覺素材原則（IP 風險）

- **不使用 Snoopy 或任何受版權保護的角色 IP**
- 「Snoopy Museum 風格」僅作為**調性參考**（卡通活潑、暖色、圓潤），不直接使用其角色
- Phase 1–2 優先用**無 IP 風險的素材**：emoji、原創 SVG、幾何裝飾、棒球物件（球、手套、球棒）
- 若需角色化 mascot，應**另行委託原創設計**，建立可長期使用的品牌資產
- 所有使用到的球隊 logo 僅用於賽事資訊展示（與現行使用方式一致），非個人化功能的裝飾素材

## 核心設計原則

黏著度的三個支柱：

1. **有理由回來**（主動提醒、個人化內容）
2. **累積感讓人捨不得離開**（打卡紀錄、徽章、年度回顧）
3. **能跟人分享**（社交功能、公開頁、預測互動）

以此為基礎分六階段推進，每階段都建立在前一階段的資料基礎上，Phase 1 的 schema 設計將決定後續所有階段的擴充彈性。

## Phase 1 — 個人化基礎（建立身份）

**目標**：讓使用者從「路人查賽程」變成「這個網站認得我」。

### 功能

- **主隊設定**：首次進站選擇，整站預設用該隊視角（戰績高亮、對戰卡排序、預設篩選）
- **場次收藏 ⭐**：未來場次加星號，可獨立檢視「我的收藏」頁籤
- **觀戰打卡 📍**：已結束場次可打卡，欄位含：
  - 座位區（內野／外野／全壘打牆等，可選）
  - 同行者（純文字，非帳號）
  - 5 星評分
  - 心得文字（支援 200 字）
  - 照片（初期存 base64，或先不支援上傳）
- **篩選記憶**：下次進站記得上次篩選條件

### 技術

- 儲存：localStorage
- Schema 必須預先帶版本號，供 Phase 4 同步時遷移：

```js
{
  version: 1,
  profile: { mainTeam: "中信兄弟" },
  favorites: ["001", "045", "112"],
  checkins: {
    "001": { seat, companions, rating, note, photo, createdAt }
  },
  lastFilter: { teams: [...], venues: [...], ... }
}
```

## Phase 2 — 成就與情感連結（給回訪理由）

**目標**：把「資料」變成「我的故事」。這階段最重要，多數規劃器死在沒做這層。

### 功能

- **球場巡禮徽章**：11 座球場收集進度（已打卡亮色、未去灰階 + 剪影）
- **現場戰績卡**：「你支持的中信兄弟，在你現場觀戰時 5 勝 3 敗（勝率 .625）」
- **Streak 指標**：連續去幾場、連勝／連敗現場見證
- **我的 2026 球季回顧頁**（獨立 URL `/me/2026`）：
  - 卡通活潑風格插畫（原創，非 Snoopy IP）+ 關鍵數字
  - 去過幾場、最常去哪座球場、現場見證的最大逆轉
  - 可截圖分享（html2canvas 或動態 og:image）
  - 年底 12 月彈出「你的年度回顧已生成」toast
- **里程碑動畫**：
  - 第一次去新球場
  - 第 10 場打卡
  - 支持球隊晉級時跳成就動畫

### 技術

- 純前端 canvas / SVG 渲染，不需後端
- 視覺風格沿用現有暖色調（赤陶紅 + 奶茶底），保持卡通活潑調性但不使用 Snoopy IP

## Phase 3 — 主動提醒（被動回流）

**目標**：使用者不需要主動想到網站，網站會提醒他。

### 功能

- **單場 + 批次 .ics 下載**：收藏清單一鍵匯出到 Google Calendar / iOS 行事曆
- **PWA 安裝**：加到手機主畫面，有 icon 有啟動畫面（原創棒球主題 splash）
- **本機通知（Notification API）**：
  - 比賽當天 10:00 推播「今晚有你收藏的場次」
  - 天氣警報：降雨率 > 60% 提前 3 小時提醒（整合現有 Open-Meteo）
- **售票開賣追蹤**：各隊主場售票時間不同，用前端排程在日期到前提醒

### 技術

- PWA manifest + Service Worker
- Notification API（需使用者授權）
- 延伸現有天氣 API

---

## Phase 4 — 跨裝置同步（黏著度瓶頸突破）

**目標**：解決「手機打卡，電腦看不到」的痛點，這是 Phase 1-3 規模化的前提。

### 功能

- **過渡方案**：匯出 / 匯入 JSON 按鈕（先解決部分痛點）
- **正式版**：Cloudflare Workers + KV（或 D1），免費額度足夠
- **登入方式**：
  - 優先 Magic Link（email 一次性登入連結，用 Resend／Postmark 免費額度）
  - 次選 GitHub OAuth（技術簡單，但目標受眾不一定有 GH 帳號）
- **隱私**：
  - 所有資料加密存
  - 使用者可一鍵刪除全部資料
  - 明確告知資料使用範圍

### 技術

- 唯一需要認真寫後端的階段，預估 1–2 週
- Cloudflare Workers 與現有架構（GitHub Pages）相容

## Phase 5 — 社交互動（帶來新用戶）

**目標**：讓現有用戶變成拉新引擎。

### 功能

- **分享篩選 URL**：篩選狀態寫入 query string，「中信兄弟 × 洲際 × 週末」一鍵複製
- **公開個人頁**：`/u/tiffany` 顯示打卡足跡（使用者可選公開／私人）
- **每日猜勝負**：
  - 當天場次開賽前可預測
  - 累積預測準確率
  - 衍生：全站預測排行榜、單場社群預測比例
- **揪團場次**：收藏場次下方「1 個朋友也收藏了這場」（需 Phase 4 登入）
- **球季挑戰**：
  - 「3 個月走訪 5 座球場」
  - 「週末戰士」
  - 公開挑戰可追蹤進度、完成徽章

## Phase 6 — 內容深度（留住硬核粉）

**目標**：到這階段已是 CPBL 球迷的主力工具，深度內容鞏固地位。

### 功能

- **球員追蹤**：關注的球員今日出賽、近況、里程碑倒數
- **對戰歷史戰績**：每張卡展開「兩隊近 10 次對戰 6-4」+ 主場優勢
- **球場深度頁**：
  - 交通（延伸現有高鐵資訊）
  - 停車、場內美食、觀戰座位建議
  - 使用者評分彙整
- **賽事記錄升級**（延伸現有功能）：結合個人打卡資料「你現場見證了這場逆轉」
- **賠率 / 進階數據**（選配）：三振率、OPS 等

## 技術架構演進

| 階段 | 儲存 | 後端 | 關鍵依賴 |
|------|------|------|---------|
| Phase 1–3 | localStorage | 無 | 現有靜態架構 |
| Phase 4 | Cloudflare KV | Workers | 登入服務（Magic Link / OAuth） |
| Phase 5–6 | KV + D1 | Workers | 關聯查詢、排行榜彙整 |

### 關鍵決策點

- **Phase 1 的資料 schema 必須預先設計好**，避免 Phase 4 遷移時阻塞
- **Phase 4 是架構轉折點**：決定是否走後端路線，以及選擇哪種登入方式
- **Phase 5–6 順序可彈性調整**：依實際使用者回饋排序

## 建議執行節奏

| 里程碑 | 涵蓋範圍 | 預估時程 | 效益 |
|--------|---------|----------|------|
| 最小可用 | Phase 1 + 2 | 3–4 週 | 顯著黏著度提升 |
| 產品感完整 | Phase 1–3 | 5–6 週 | 具備 PWA + 日曆匯出 |
| 規模化轉折 | Phase 4 | +1–2 週 | 跨裝置同步，可做社群功能 |
| 社群規模 | Phase 5 | +2–3 週 | 帶來新用戶 |
| 深度專業 | Phase 6 | 持續迭代 | 鞏固硬核粉市場 |

### 優先執行建議

1. 先完成 Phase 1 + Phase 2，這兩階段成本低、效益高，且不需任何後端
2. 觀察 Phase 2 的使用者回訪率，決定是否推進 Phase 3
3. Phase 4 前建議先做 JSON 匯出 / 匯入，作為過渡方案驗證需求
4. Phase 5 與 6 根據使用者實際行為資料決定順序
